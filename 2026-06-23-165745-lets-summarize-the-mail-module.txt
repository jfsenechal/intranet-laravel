  ‚ö†ÔłŹ Discrepancies / things that don't match your summary

  1. "A letter has one or more departments" ‚ÄĒ no.
  A letter has exactly one department. The many relationship is services()
  (incoming_mail_service, with is_primary). You're likely conflating services
  with departments.

  2. INDEX role does NOT actually gate the index page.
  The courrier-index gate (which checks the *_INDEX roles) is only used in
  SendIncomingMailNotificationJob::recipientHasIndexRole() to decide who gets
  notified. The IncomingMailResource has no canAccess/canViewAny override, and
  IncomingMailPolicy::viewAny() returns true for everyone. So index access is
  governed by panel access (canAccessPanel() returns true for all users), not by
  *_INDEX.

  3. INDEX users are NOT filtered by department.
  DepartmentScope derives departments from getCourrierDepartments(), which only
  looks at ADMIN roles (RolesEnum::getAdminRoles()). A user with only an *_INDEX
  (or *_READ) role gets [] departments ‚Üí the scope adds no where clause ‚Üí they
  would see all departments' mail in the list, unscoped. This is the opposite of
  "filtered by department," and is a likely security gap.

  4. The *_READ roles are dead code.
  ROLE_INDICATEUR_*_READ exist in the enum but are referenced nowhere except a
  commented-out courrierDepartment() method in User.php and tests asserting
  getDepartment() is null. There is no "full read based on READ role" behavior
  implemented.

  5. Attachment download has no authorization.
  AttachmentController::download() / previewStored() (route
  courrier.attachments.download) only check that the file exists ‚ÄĒ no
  recipient/service policy check. The IMAP show/preview routes likewise have
  only web+auth. So any authenticated user can download any attachment by ID,
  regardless of being a recipient or service member. The view-page rule from
  point above is not carried into the download route.
