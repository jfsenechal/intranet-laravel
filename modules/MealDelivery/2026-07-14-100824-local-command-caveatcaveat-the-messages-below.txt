  No ‚ÄĒ the new MealDelivery module does not implement this legacy logic

  The three behaviors in the selected legacy code all relate to the physical
  order sheet (la "feuille") the delivery driver hands out or takes back, plus
  the disposable-container marker. None of the automatic computations survive in
  the new module.

  The legacy logic (from ClientManager::setRepas + RepasManager)

  Three per-meal flags computed automatically when building the driver's route
  sheet:

  1. isLastRepasAndCloseClient ‚Üí disposable recipient (last meal of a closing
  client)
  2. takeBackSheet ‚Üí take the sheet back if it's the last meal, the client has
  no order next week, AND it's Wednesday or later
  (isEqualOrGreaterThanWednesday)
  3. isFirstRepasOfWeek ‚Üí give a new sheet on the first meal of the week

  What the new module has instead

  - A single manual is_last_meal boolean on the Order model (Order.php:14),
  toggled in OrderForm.php:46 with helper text "Si oui, prendre feuille nouvelle
  commande". It's shown as a column/filter/infolist icon, but it's
  operator-set, not computed.
  - RouteSheetsAggregator (the new equivalent of ClientManager::setRepas) builds
  route-sheet rows with quantities, diets, addresses and notes ‚ÄĒ but computes
  none of: first-meal-of-week, take-back-sheet, Wednesday cutoff, or disposable
  recipient.
  - The route-sheet blades (_route-sheet-table, route-sheet-pdf, route-sheets)
  display no "nouvelle feuille" / "reprendre feuille" / "jetable" markers.

  Bottom line

  The automatic sheet-management logic was not ported. It was replaced by one
  manual is_last_meal toggle, which only loosely covers case #2/#3 (and only if
  a human remembers to set it). The "first meal of week ‚Üí new sheet", the "no
  order next week" lookahead, the Wednesday cutoff, and the disposable-recipient
  flag have no equivalent in the current module.
