  Summary

  I added Share and Ask actions to the courrier's ViewIncomingMail page (modules
  /Courrier/src/Filament/Resources/IncomingMails/Pages/ViewIncomingMail.php).

  Share (share action)
  - Visible only when the courrier has an attachment and the current user
  can('download', …) it.
  - Modal: a searchable, bulk-toggleable CheckboxList of all recipients with an
  email + a Message textarea.
  - On submit, sends ShareIncomingMail to each selected recipient — carrying the
  courrier details and the attachment(s).

  Ask (ask action)
  - Visible only when the courrier has an attachment and the current user cannot
  download it (the complementary condition).
  - Modal: a CheckboxList pre-filled with the department's attachment readers —
  recipients flagged receives_attachments linked to a service of the courrier's
  department. Options are fixed to that set, so the asker can deselect but not
  add anyone. Plus a Message textarea.
  - On submit, sends AskAttachment to each selected reader — courrier details +
  a link to the view, no attachment, naming the asker (reply-to set to their
  email) so they can hit Share to send it back.

  New files
  - src/Mail/ShareIncomingMail.php, src/Mail/AskAttachment.php
  - resources/views/mail/share-incoming-mail.blade.php,
  resources/views/mail/ask-attachment.blade.php
  - tests/Feature/Courrier/IncomingMailShareAskTest.php (3 tests, 25 assertions)

  Changed
  - src/Repository/RecipientRepository.php — added getShareOptions() (all
  recipients) and getAttachmentReaderOptions(?string $department) (dept readers
  with receives_attachments).

  Decisions followed your answers: share list = all recipients; ask list =
  department recipients with receives_attachments; no persistence — the asker is
  named in the email only.

  Two notes worth flagging:
  - Emails go to Recipient records (they hold the emails and mirror the
  notification system). If the asking user isn't themselves a recipient of the
  courrier's department, they won't appear in a reader's Share list — the "name
  asker in email only" option you picked accepts that trade-off.
  - Mailables are sent synchronously inside the action. If a share with large
  attachments feels slow, we can move it to a queued job later.

  Pint is clean. Note: npm run build isn't needed (no frontend asset changes),
  but a queue worker should be running if you later queue these.
